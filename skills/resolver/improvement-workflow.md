# Resolver改进工作流

> 当用户表示不满意时，触发本改进流程

---

## 一、改进触发条件

### 1.1 显式触发

| 用户表达 | 含义 |
|----------|------|
| "不对" | 结果方向错误 |
| "不是这样" | 格式/结构不符合预期 |
| "不满意" | 整体质量不达标 |
| "重来" | 要求重新执行 |

### 1.2 隐式触发

| 判定条件 | 说明 |
|----------|------|
| 用户修改超过3次 | 反复调整说明方向不明 |
| 任务耗时超过预期2倍 | 执行效率问题 |
| 多次触发重试 | 可能存在系统性问题 |

## 二、改进流程

### 2.1 标准改进循环

```
┌─────────────────────────────────────────────────────────────┐
│                    改进工作流（5步）                        │
├─────────────────────────────────────────────────────────────┤
│  Step 1: 理解问题                                            │
│          - 用户想要什么？                                     │
│          - 当前结果差在哪里？                                 │
│          - 有没有参考示例？                                   │
│                     ↓                                        │
│  Step 2: 分析根因                                            │
│          - 是技能description问题？                          │
│          - 是路由匹配问题？                                  │
│          - 是执行逻辑问题？                                  │
│                     ↓                                        │
│  Step 3: 制定改进方案                                        │
│          - 修改description？                                  │
│          - 添加新触发词？                                     │
│          - 优化执行流程？                                    │
│                     ↓                                        │
│  Step 4: 执行改进                                            │
│          - 更新配置文件                                       │
│          - 测试验证                                          │
│          - 记录改进                                          │
│                     ↓                                        │
│  Step 5: 反馈用户                                            │
│          - 说明改了什么                                      │
│          - 请用户确认                                        │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 快速改进（针对单一问题）

```python
quick_fix = {
    "trigger": "用户指出具体问题",
    "steps": [
        "1. 复述问题确认理解",
        "2. 立即修改",
        "3. 重新执行",
        "4. 请用户确认"
    ],
    "max_iterations": 2
}
```

## 三、问题分类与处理

### 3.1 技能Description问题

| 问题 | 表现 | 处理方式 |
|------|------|----------|
| 触发词不全面 | 技能未被匹配 | 添加新触发词到description |
| 描述不准确 | 匹配但结果不对 | 修正description内容 |
| 缺少使用场景 | 用户不知道能用 | 在description中添加场景说明 |

**处理示例**：
```yaml
# 问题：用户说"帮我分析观点"但 diarization 未被匹配
# 原因：description中缺少"分析观点"触发词

# 修复：将
description: "...触发词：diarization、结构化档案、抽取档案"

# 改为
description: "...触发词：diarization、结构化档案、抽取档案、专家观点、观点分析"
```

### 3.2 路由配置问题

| 问题 | 表现 | 处理方式 |
|------|------|----------|
| 关键词冲突 | 匹配到错误技能 | 调整优先级 |
| 缺少映射 | 无技能被匹配 | 添加新映射或调用skill-assistant |

### 3.3 执行逻辑问题

| 问题 | 表现 | 处理方式 |
|------|------|----------|
| 参数理解偏差 | 参数传递错误 | 优化SKILL.md中的参数说明 |
| 流程不合理 | 执行顺序错误 | 调整执行流程 |
| 缺少容错 | 遇到边缘情况失败 | 添加错误处理 |

## 五、持续优化机制

### 5.1 周期性检查

| 周期 | 内容 |
|------|------|
| 每日 | 检查失败日志，分析高频失败 |
| 每周 | 更新触发词库，补充新场景 |
| 每月 | 评估技能识别准确率，优化路由规则 |

### 5.2 用户反馈收集

```python
feedback_collection = {
    "positive": ["满意", "很好", "对", "可以"],
    "negative": ["不对", "不是", "不满意", "重来"],
    "neutral": ["嗯", "好的", "知道了"]
}
```

### 5.3 自动优化规则

```
IF 用户反馈 negative_count > 2 AND same_skill:
    → 触发技能自进化
    → 更新description和路由配置
    → 记录改进日志
```

## 六、最佳实践

1. **优先理解用户意图**：不要急于修改，先确认理解正确
2. **小步快跑**：每次改进一个点，快速验证
3. **保留备份**：修改前备份原文件
4. **记录改进**：便于后续分析和复用
5. **用户确认**：改进后必须请用户确认
